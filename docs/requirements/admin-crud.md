# Feature: 管理者CRUD

## 目的
- 運用者が `Admin` アカウントを作成・参照・更新・削除できるようにする。
- 既存の Devise 認証（`Admin`）を前提に、管理対象としての CRUD を提供する。

## スコープ（チェックリスト）

- [ ] Admin 削除（論理削除）
- [ ] 認可: 非ログイン/権限外ユーザーのアクセス拒否
- [ ] 認可: `super_admin` / `operator` の権限制御
- [ ] 認可: 作成・更新・削除は `super_admin` のみ実行可能
- [ ] 権限: 新規作成された管理者のデフォルトロールは `operator`
- [ ] 制約: 自分自身のアカウントは削除不可
- [ ] 一覧: `ransack` で `id` / `email` 検索

- [ ] 一覧: `pagy` で 30 件/ページ
- [ ] 監査: 作成・更新・削除の操作ログ方針を定義

## Gherkin
```gherkin
Feature: 管理者情報を安全に管理する
  認証済み管理者として
  管理者アカウントの作成・参照・更新・削除を行いたい

  Background:
    Given 認証済みの管理者がログインしている

  Scenario: 管理者一覧を表示する
    Given 複数の管理者が登録済みである
    When ユーザーが管理者一覧ページを開く
    Then 管理者の一覧が1ページ30件で表示される
    And 各行にメールアドレスと状態情報が表示される
    And 一覧は `id desc` がデフォルトで適用される

  Scenario: 管理者詳細を表示する
    Given 管理者レコードが存在する
    When ユーザーが管理者詳細ページを開く
    Then ポリシーで許可された管理者属性が表示される

  Scenario: 管理者を新規作成する（正常系）
    Given ユーザーが管理者新規作成ページを開いている
    And 操作ユーザーは `super_admin` である
    When ユーザーが有効な管理者属性を送信する
    Then 新しい管理者が作成される
    And 新しい管理者のロールは `operator` になる
    And 成功メッセージが表示される

  Scenario: 管理者の新規作成に失敗する（異常系）
    Given ユーザーが管理者新規作成ページを開いている
    And 操作ユーザーは `super_admin` である
    When ユーザーが無効な管理者属性を送信する
    Then 管理者は作成されない
    And バリデーションエラーが表示される

  Scenario: 管理者を更新する（正常系）
    Given 管理者レコードが存在する
    And 操作ユーザーは `super_admin` である
    When ユーザーが編集ページで有効な変更を送信する
    Then 管理者情報が更新される
    And 成功メッセージが表示される

  Scenario: 管理者の更新に失敗する（異常系）
    Given 管理者レコードが存在する
    And 操作ユーザーは `super_admin` である
    When ユーザーが編集ページで無効な変更を送信する
    Then 管理者情報は更新されない
    And バリデーションエラーが表示される

  Scenario: 管理者を論理削除する
    Given 削除対象の管理者レコードが存在する
    And 操作ユーザーは `super_admin` である
    When ユーザーが管理者を削除する
    Then 管理者レコードは論理削除状態になる
    And 管理者一覧のデフォルト表示対象から除外される
    And 成功メッセージが表示される

  Scenario: 認可ガードを適用する
    Given 非ログインユーザーが管理者CRUDのエンドポイントにアクセスする
    Then サインインページにリダイレクトされる
    When 認証済みだが未許可のユーザーが管理者CRUDのエンドポイントにアクセスする
    Then 禁止レスポンスで拒否される

  Scenario: 自分自身のアカウント削除を拒否する
    Given ログイン中の管理者自身のレコードが存在する
    When ユーザーが自分自身を削除しようとする
    Then 削除は実行されない
    And エラーメッセージが表示される

  Scenario: ロール別権限制御を適用する
    Given `super_admin` と `operator` のロールが定義されている
    When `operator` が管理者作成・更新・削除を実行する
    Then 操作は拒否される
    When `super_admin` が管理者作成・更新・削除を実行する
    Then 操作は許可される

  Scenario: 一覧でIDとメールアドレス検索を行う
    Given 複数の管理者が登録済みである
    When ユーザーが `ransack` を使って `id` または `email` で検索する
    Then 条件に一致する管理者だけが一覧に表示される

  Scenario: 一覧のソートとページネーションを切り替える
    Given 30件を超える管理者が登録済みである
    When ユーザーが一覧を `id asc` に切り替える
    Then 一覧は `id` 昇順で表示される
    And ページネーションで次ページへ遷移できる
```

## 影響範囲メモ（実装前提）
- モデル: `Admin`（既存 Devise モデル）
- ルーティング: 管理用 `admins` リソース
- コントローラ/ビュー: 管理画面用 CRUD
- テスト: request spec を中心に認証・認可・バリデーション・CRUD 成否を検証

## リスク
- High: `Admin` はログイン主体のため、自己削除・他者削除の制約を誤ると運用不能化リスクがある。
- Medium: Devise パラメータと独自 CRUD の strong parameters が競合する可能性がある。
- Medium: 論理削除導入時に Devise 認証・ユニーク制約・既存スコープの整合性を崩す可能性がある。

## 確定事項
- 削除は論理削除を採用する。
- 作成・更新・削除は `super_admin` のみ実行可能とする。
- 新規作成された管理者のデフォルトロールは `operator` とする。
- 自分自身のアカウント削除は不可とする。
- 権限レベルとして `super_admin` / `operator` を導入する。
- 一覧は `ransack` を利用し、`id` / `email` で検索可能とする。
- 一覧は `id desc` をデフォルトにし、`id asc` へ切り替え可能とする。
- ページネーションは `pagy` を利用し、30件/ページとする。
